# Design Compare（多方案对比工作流）

用户要"多对比"时，**给可渲染的真东西，不要给文字描述**。一次产出 2–4 个明显不同的变体，并列呈现，让用户用眼睛选。

## 一、先确定对比的"轴"

一次只沿**一条轴**变化，其余固定，否则无法归因谁好谁坏。

- **轴 A · 跨审美锚点**（最常用）：同一个页面，切 `themes.css` 的 4 套主题（Wise / Mercury / Revolut / Stripe），比气质。用 `data-theme` 切换器实时翻，或分别截图。
- **轴 B · 跨布局构图**：锁定一套主题，比 2–4 种信息架构/Hero 构图/密度（如结汇报价卡的三种排布）。
- **轴 C · 跨局部**：锁定页面，比某个关键组件的几种做法（卡片视觉、状态呈现、表格密度）。

**别做 3 个几乎一样的变体**——每个都必须是真实的选择，且都避开 `frontend-design` 里那三张 AI 默认脸。

## 二、流程

1. **选屏 + 选轴**：挑一个有代表性的屏（推荐结汇报价卡 / 收单仪表盘 / 发卡卡片墙），一句话说清对比什么。
2. **出 token 计划**：每个变体先按 `frontend-design` 的两遍法写紧凑 token 计划，注明引用了哪套锚点、signature 是什么。
3. **构建变体**：真实渲染。跨主题对比时，同一份组件代码 + 不同 `data-theme` 即可（这正是 `themes.css` 多主题的价值）。
4. **截图**：能截就截（`chrome-devtools` 或 `playwright` MCP）。桌面 + 移动各一张。一图胜千 token。
5. **并列呈现 + 权衡表**：每个变体给「适合场景 / 优点 / 代价」一句话，别只夸不说代价。
6. **用户挑** → **落库**：把选定方向写进 `docs/DESIGN.md`（见下），后续所有屏遵守它。

## 三、对比呈现格式

放截图缩略 + 下面这张表：

| 变体 | 锚点 / 方向 | Signature | 适合 | 代价 |
|---|---|---|---|---|
| A | Wise 亮绿 | 到手金额巨大化 + 费用全展开 | C 端汇款、信任优先 | 信息密度低，不适合后台 |
| B | Mercury 极简 | 冷静表格 + 单钴蓝强调 | 发卡/收单后台 | 偏冷，C 端亲和力弱 |
| … | … | … | … | … |

## 四、落库：docs/DESIGN.md（防漂移）

用户一旦选定，立刻写入 `docs/DESIGN.md`，让后续会话/页面不跑偏：

```markdown
# 设计决策记录（Design Decisions）
- **主锚点**：Wise（结汇/前台）；后台用 Mercury。data-theme 默认 "wise"。
- **品牌色**：--brand #9fe870 / --brand-strong #163300。
- **圆角/密度**：前台 14px 舒适；后台 8px 紧凑。
- **字体**：Inter + PingFang/思源；金额 tabular-nums。
- **已定组件**：报价卡=变体A；状态=Badge+文字+下一步。
- **待定**：发卡前台是否单独用 Revolut 风（YYYY-MM-DD 复盘）。
- **变更记录**：<日期> 把 X 从 A 改为 B，因为……
```

## 五、对比前自查
- [ ] 只变了一条轴，其余固定。
- [ ] 每个变体都是真实选择，都避开三张 AI 默认脸，彼此差异明显。
- [ ] 有截图（桌面+移动），不是纯口头描述。
- [ ] 权衡表说了代价，不只说优点。
- [ ] 用户选定后已写入 `docs/DESIGN.md`。
