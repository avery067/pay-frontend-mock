# Fintech Payment UI（跨境支付设计核心）

做**结汇 / 发卡 / 收单**相关界面时读它。它把三样东西绑在一起：**选一套审美锚点** → **套用支付域模式** → **守住双语排版**。先读 `frontend-design`（地基），再用本 skill。

## 一、四套审美锚点（可切换主题）

全部 token 定义在同目录 **`themes.css`**（可直接拖进工程，Tailwind v4 + shadcn 变量约定）。每套锚点把配色/圆角/阴影/字体/密度锁死，**选定就别串味**（不要"Wise 加一点 Stripe 边"这种杂交）。

| 锚点 | 品牌色 | 气质 / 圆角 / 密度 | 最适合的业务面 |
|---|---|---|---|
| **Wise**（旗舰，默认） | 亮绿 `#9FE870` + 森林绿 `#163300` | 透明·亲和·直白 / 圆润 14px / 舒适 | 结汇报价、汇率、到账状态、C 端汇款 |
| **Mercury·Ramp** | 近黑白灰 + 冷静钴蓝 `#2563EB` | 极简·专业·高信息密度 / 8px / 紧凑 | 发卡后台、收单对账、财务台账 |
| **Revolut·Monzo** | 珊瑚 `#FF3B6B` 单色主导 | 年轻·大胆·活力 / 圆润 20px / 舒适 | 持卡人前台、卡片管理、消费流水 |
| **Stripe·Linear** | 靛蓝 `#635BFF` + 渐变 | 精工·科技·深色优先 / 8px / 中等 | 官网、落地页、开发者/API 文档 |

**怎么用**：
- 需要**一次看多套**做对比 → 走 `design-compare` skill，用 `data-theme` 切换器实时翻。
- 已定方向 → 用该锚点的 token，界面内保持一致；决策写进 `docs/DESIGN.md`（见 `design-compare` / `mock-scaffold`）。
- 想加第 5 套自定义主题 → 复制 `themes.css` 里任一 `[data-theme]` 块改 token，保持同一份变量契约。

技术上：每套主题是一个 `[data-theme="wise"] { --primary: …; … }` 块，覆盖 shadcn 标准变量 + 本项目新增的 fintech 语义变量（`--success/--warning/--danger/--info`、金额涨跌 `--pos/--neg`、渐变 `--gradient-brand`）。`.dark` 与 `[data-theme]` 可组合。细节见 `themes.css` 顶部注释。

## 二、支付域 UX 模式

完整模式库见同目录 **`patterns.md`**，覆盖：

- **结汇（FX / Settlement）**：报价卡（汇率 + 点差 + 费用全透明，Wise 式）、金额输入与币种选择、汇率锁定倒计时、结算时间线（发起→处理中→已结算）、合规/KYC 闸口、回执。
- **发卡（Card Issuing）**：虚拟卡视觉组件、卡片列表、开卡流程、额度与消费控制、冻结/注销、卡交易流水、卡 BIN/项目信息。
- **收单（Acquiring）**：商户仪表盘、交易高密度表格、结算与对账（reconciliation）、退款/争议、清算周期与账期。

三条线密度不同，锚点也应随之切换：**结汇/前台偏 Wise 舒适**，**发卡/收单后台偏 Mercury·Ramp 高密度**。

## 三、金额与状态（贯穿所有界面）

摘要如下，细则见 `patterns.md`：

- **金额**：一律用等宽数字（`tabular-nums`）+ 拉丁字体，千分位分组，货币代码/符号位置固定；正负用 `--pos/--neg`（**不要**只靠红绿，同时给符号 `+ / −`）。大额用空格分隔而非挤在一起。
- **状态语义**（映射到 token，勿自造颜色）：
  - 成功/已结算/已到账 → `--success`
  - 处理中/待处理/审核中 → `--warning`
  - 失败/被拒/超时 → `--danger`
  - 信息/提示 → `--info`
  - 中性/草稿/已关闭 → `--muted-foreground`
- 状态用 **Badge + 文字**，不要只用色点；关键状态给可点击的下一步（Wise 式"每条日志都可操作"）。

## 四、双语排版

细则见同目录 **`typography-bilingual.md`**。要点：

- 字体栈把**拉丁字体放前面**（`Inter` → `PingFang SC` → `Source Han Sans SC`），让 ASCII 数字/字母走 Inter、汉字自然回退到中文字体；
- **金额、编号、单位、日期一律拉丁 + `tnum`**；
- CJK 与拉丁/数字之间加"盘古之白"（细空隙），行高比纯拉丁更松（正文 1.6–1.75）；
- **别用微软雅黑当主字体**，别给 CJK 加 faux-bold，别对 CJK 做负字距——这些是"土气"的主要来源。

## 五、fintech 的信任线索（做对，别做成 slop）

支付产品的高级感很大程度来自"可信"，而非装饰：

- **费用与汇率前置透明**（Wise 的核心）：把点差、手续费、到账时间、最终到手金额摆在最显眼处，别藏进"详情"。**结构性诚实 > 装饰性认证徽章。**
- **状态可追溯**：每笔交易/结算可展开看轨迹、可复算、可定位问题。给真实的下一步动作，别给"格式化得很高级的噪音"。
- **安全线索要真**：用真实的合规信息（KYC 状态、限额、风控提示）而非 `🔒 SECURE OPERATOR` 这类填充文案。徽章/锁图标只在承载真实信息时才用。
- **敏感信息脱敏**：卡号 `•••• 4242`、账号中段打码、金额可一键隐藏（前台）。

## 六、发货前检查清单

- [ ] 只用了**一套锚点**，token 没有串味（Wise 里没混进 Stripe 渐变，Mercury 里没混进大圆角）。
- [ ] 金额/数字全部 `tabular-nums` + 拉丁字体，正负有符号不只靠颜色。
- [ ] 状态颜色全部来自语义 token，没有硬编码的红绿。
- [ ] 费用/汇率/到账透明前置；无填充式"安全"文案；敏感信息已脱敏。
- [ ] 中西混排：数字走拉丁、CJK 未负字距、CJK–拉丁间有空隙、行高够松。
- [ ] 质量底线：响应式到移动端、键盘焦点可见、`prefers-reduced-motion` 已尊重、深色模式可用。
- [ ] 没有假数据冒充真实数据；示例数据一眼能看出是示例（见 `mock-scaffold` 假数据规范）。
